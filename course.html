Chapter 1

What Is Web.Serial?

Imagine plugging a hardware device (like an Arduino) into your computer and talking to it directly from your web browser — no app download needed.

🔌 The Problem It Solves

Normally, to communicate with hardware devices over a serial port (USB cable), you need to install desktop software like PuTTY, the Arduino IDE, or screen. Web.Serial replaces all of that — it runs entirely in your browser.

  • Works in Chrome, Edge, and Opera (desktop)
  • No installation, no drivers, no build step
  • Settings auto-save so you can pick up where you left off
  • Includes a real-time data plotter for graphing sensor data
  • Works offline once loaded (it's a Progressive Web App)

🧩 How It's Built

The entire app is plain HTML, CSS, and JavaScript — no React, no Vue, no build tools. Six JavaScript files, each responsible for one job:

File What It Does
config.js Remembers your settings (baud rate, theme, etc.)
serial-manager.js Talks to the USB device via Web Serial API
terminal-renderer.js Displays text on screen, including colored text
input-handler.js Handles keyboard input in two different modes
plotter.js Draws real-time line graphs from serial data
app.js Wires everything together and manages the UI

📡 Events: The Secret Sauce

When the serial port receives data, it doesn't call the terminal directly. Instead, it shouts serial:data to the whole page. Any module that cares about serial data can listen:

// In serial-manager.js — Broadcasting const text = this.decoder.decode(value); this.emit("data", text); // emit() creates a custom event: emit(eventName, detail) { window.dispatchEvent( new CustomEvent( `serial:${eventName}`, { detail } ) ); }

Think of it like a radio broadcast.

Line 1-2 The serial module decodes raw bytes into text and calls emit("data", text).

Line 5-11 emit() creates a Custom Event named "serial:data" and dispatches it on the window object.

Any module that wants to react to incoming data just says window.addEventListener("serial:data", ...). Both the terminal and the plotter do this independently.

Why events? If the serial module had to know about the terminal AND the plotter AND any future module, the code would become tangled. Events keep everything loosely coupled — you can add or remove listeners without changing the broadcaster.

🗂️ The Event Catalog

Event Name Fired When… Data Carried
serial:connected Device is connected USB vendor/product IDs
serial:data Bytes arrive from device Decoded text string
serial:disconnected Device is disconnected —
serial:error Something goes wrong Error object
serial:stats Bytes counted {rx, tx} byte counts
config:changed A setting changes {key, value}

Chapter 3

Remembering Your Settings

Close the tab, come back tomorrow, and everything is exactly how you left it. Here's how that works.

config.js — The Memory Keeper

This module uses localStorage — a built-in browser feature that stores data permanently on your computer (like a tiny invisible notepad).

const CONFIG_KEY = "web_serial_terminal_config"; const DEFAULT_CONFIG = { baudRate: "115200", dataBits: "8", stopBits: "1", parity: "none", flowControl: "none", terminalMode: "modern", lineEnding: "\r\n", localEcho: true, theme: "dark", quickCommands: [ /* ... */ ], history: [], };

A dictionary of defaults.

Line 1 The storage key — all settings live under one name in localStorage.

Lines 3–15 Every setting has a sensible default. If you've never used the app before, these are what you get.

Settings include serial port parameters (baud rate, etc.), UI preferences (theme, mode), and even your saved command shortcuts.

💾 Load → Merge → Save

The clever part: when loading saved settings, the code merges them with the defaults. This means if a new version adds a setting, your old save file won't break — the new setting just gets its default value.

loadConfig() { const saved = localStorage.getItem(CONFIG_KEY); if (saved) { const parsed = JSON.parse(saved); return this.mergeDeep(DEFAULT_CONFIG, parsed); } return structuredClone(DEFAULT_CONFIG); } set(key, value) { this.settings[key] = value; this.saveConfig(); window.dispatchEvent( new CustomEvent("config:changed", { detail: { key, value } }) ); }

loadConfig() Tries to read from localStorage. If found, it deeply merges old + new defaults so nothing is lost.

set() When you change a setting:

  1. Updates the in-memory copy
  2. Writes to localStorage immediately
  3. Fires a config:changed event so the UI updates

This three-step pattern is used everywhere — it's what makes the app feel instantly responsive.

Try it yourself: Open the app, change a setting, then type this in DevTools console: JSON.parse(localStorage.getItem('web_serial_terminal_config')). You'll see all your settings stored as JSON.

serial-manager.js — The USB Whisperer

async connect(options) { // Step 1: Ask user to pick a device this.port = await navigator.serial.requestPort(); // Step 2: Open with serial settings await this.port.open({ baudRate: parseInt(options.baudRate), dataBits: parseInt(options.dataBits), stopBits: parseInt(options.stopBits), parity: options.parity, flowControl: options.flowControl, }); // Step 3: Start the infinite read loop this.keepReading = true; this.startReading(); // Step 4: Tell everyone we're connected this.emit("connected", this.port.getInfo()); }

Four steps to connection:

Step 1 The browser pops up a device picker — the user must explicitly choose which device to allow. This is a security feature.

Step 2 Opens the serial port with the configured baud rate and other settings. Both sides (browser + device) must agree on these values.

Step 3 Starts an infinite loop that continuously reads bytes from the device.

Step 4 Broadcasts a serial:connected event so the UI and other modules can react.

🔁 The Infinite Read Loop

Once connected, the serial manager enters a loop that never stops (until you disconnect). Every time bytes arrive, it decodes them and broadcasts:

async startReading() { while (this.port.readable && this.keepReading) { this.reader = this.port.readable.getReader(); try { while (true) { const { value, done } = await this.reader.read(); if (done) break; this.stats.rx += value.byteLength; const text = this.decoder.decode(value); this.emit("data", text); } } finally { this.reader.releaseLock(); } } }

An "await" loop: reader.read() pauses until bytes arrive. No busy-waiting, no CPU waste.

Line 6 Each read() returns a Uint8Array of bytes and a done flag.

Line 9 Counts received bytes for the RX counter in the footer.

Line 10-11 Decodes bytes to text (UTF-8), then broadcasts via serial:data event.

Line 13 finally always releases the reader lock, even if an error occurs — preventing the port from getting stuck.

Baud rate mismatch = garbage! If the app uses 115200 baud but the device expects 9600, you'll see random characters instead of text. Both sides must agree on the same speed.

Chapter 5

Displaying Colored Text

When a device sends "Hello" in red, it doesn't send the word "red" — it sends invisible control codes. Here's how the terminal understands them.

terminal-renderer.js — The Screen Painter

The terminal has four types of output, each styled differently:

Type Looks Like Speed
'normal' Plain text, ANSI colors parsed Batched (60fps)
'system' [SYSTEM] prefix, muted style Immediate
'error' [ERROR] prefix, red style Immediate
'echo' Orange prompt style Immediate

⚡ Smart Batching for Speed

Serial devices can send thousands of tiny data chunks per second. Updating the screen for every single chunk would freeze the browser. The solution: batch rendering.

print(text, type = "normal") { if (type !== "normal") { // System/error/echo: render NOW this.flushPending(); /* ... create DOM element immediately ... */ return; } // Normal data: collect and wait this.pendingText += text; if (!this.renderTimeout) { this.renderTimeout = setTimeout(() => { this.renderTimeout = null; this.flushPending(); }, 16); // ~60fps } }

Two fast lanes:

Lines 2-7 System messages (like "Connected!") render immediately — they're rare and important.

Lines 10-17 Normal serial data piles up in pendingText. A timer fires every 16ms (~60fps) and dumps everything to screen at once.

If 100 tiny packets arrive in 16ms, the browser only does one DOM update instead of 100. This keeps scrolling smooth even at high data rates.

🎨 ANSI Color Codes — The Invisible Paint

ANSI escape codes are invisible character sequences that tell a terminal "change the color now." They look like ESC[31m (red text) or ESC[0m (reset to default).

ansiCodeToCss(code) { if (code === 1) return "font-weight: bold; "; if (code === 3) return "font-style: italic; "; if (code === 4) return "text-decoration: underline; "; if (code >= 30 && code <= 37) { const colors = [ "black","red","green","yellow", "blue","magenta","cyan","white" ]; return `color: var(--ansi-${colors[code-30]}); `; } }

Translating numbers to colors.

Codes 1-4 Style modifiers: bold, italic, underline.

Codes 30-37 The standard 8 foreground colors. Code 31 = red, 32 = green, etc.

Instead of hardcoding hex colors, it uses CSS variables like --ansi-red. This means the colors automatically adapt to the current theme (light or dark).

Buffer limit: The terminal caps at 5,000 lines. Once exceeded, it removes the oldest third of content to prevent the browser from slowing down with a massive DOM.

Two Ways to Type

The app offers two input experiences: Modern mode (like a chat app) and Traditional mode (like a real terminal).

input-handler.js — Mode Switcher

Feature Modern Mode Traditional Mode
Input area Separate text box + Send button Type directly into the terminal
When data is sent On Enter press (whole line) Each keystroke, immediately
Cursor In the text box Blinking block in terminal output
Best for Command-response devices Interactive shells (Linux, REPL)
History ↑/↓ arrows recall past commands Not available

💬 Modern Mode — Type, Then Send

sendModern() { const text = this.inputElement.value; if (!text) return; // Save to history for ↑/↓ recall window.Config.addHistory(text); this.historyIndex = -1; // Append line ending (e.g., \r\n) const payload = text + window.Config.get("lineEnding"); this.transmit(payload); // Show what we sent (local echo) if (window.Config.get("localEcho")) { window.Terminal.print(payload, "echo"); } this.inputElement.value = ""; }

What happens when you hit Enter:

Line 5-6 The command is saved to history (max 50 entries) so you can recall it with ↑/↓ arrows.

Line 9-10 A line ending (\r\n by default) is appended — most devices expect this to know "the command is complete."

Line 13-15 Local echo — if enabled, your own text appears in the terminal so you can see what you sent. Some devices echo back themselves; others don't.

Line 17 Clears the input box for the next command.

⌨️ Traditional Mode — Live Keystrokes

In traditional mode, each key you press is sent immediately — no Enter needed. This is how classic terminals work:

handleTraditionalInput(e) { // Regular character → send immediately if (e.key.length === 1) { this.sendKeystroke(e.key); } // Enter → send the line ending else if (e.key === "Enter") { this.sendKeystroke( window.Config.get("lineEnding") ); } // Backspace → send DEL byte (0x7F) else if (e.key === "Backspace") { this.sendControlSequence(0x7F); } }

Every key = one serial transmission.

Line 3-4 Single printable characters (letters, numbers, symbols) are sent one at a time.

Line 8-11 Enter sends the configured line ending sequence.

Line 15-16 Backspace sends the DEL byte (0x7F) — this is the standard "erase" signal in serial communication.

Ctrl+C (0x03) and Ctrl+Z (0x1A) are also handled for interrupting running programs.

Chapter 7

Graphing Data In Real Time

Plug in a temperature sensor and watch the line graph draw itself live — the plotter turns serial text into visual charts.

plotter.js — The Visualizer

The plotter listens to the same serial:data event as the terminal. It expects data in a simple format:

// Format 1: Labeled values temperature:23.5,humidity:68.2 // Format 2: Just numbers (auto-named Channel 1, 2, etc.) 23.5,68.2,1013.4 // Each line = one data point for all channels

📥 How Data Gets Parsed

Serial data arrives in unpredictable chunks (not neat lines). The plotter handles this gracefully:

ingestChunk(chunk) { // On first connect, skip partial line if (this.awaitingFrameSync) { const idx = chunk.indexOf("\n"); if (idx === -1) return; chunk = chunk.slice(idx + 1); this.awaitingFrameSync = false; } // Join with leftover from last chunk const merged = this.bufferRemainder + chunk; const lines = merged.split(/\r?\n/); // Last piece might be incomplete this.bufferRemainder = lines.pop(); // Process complete lines lines.forEach(line => this.parseAndIngestLine(line)); }

Handling messy, chunked data.

Lines 2-8 Frame sync: When first connected, the first chunk might be half a line (e.g., "ure:23\n"). The plotter discards everything before the first newline.

Lines 11-12 Joins the remainder from the previous chunk with the new chunk, then splits by newlines.

Line 15 The last element after splitting might be an incomplete line, so it's saved for next time.

Line 18 Only complete lines get parsed into channel data points.

🖌️ Drawing the Chart

The plotter uses an HTML <canvas> element and draws everything from scratch 60 times per second — but only when data changes (dirty flag optimization).

  • Pan & Zoom: Scroll wheel zooms X axis, Ctrl+scroll zooms Y axis, drag to pan
  • Auto-follow: Chart scrolls with live data. Drag to detach; double-click to snap back
  • Multi-channel: Up to 8 color-coded channels with toggle-able legend
  • Interpolation: Toggle between stepped and smooth line drawing
  • Crosshair tooltip: Hover to see exact values at any point
Max samples: The plotter keeps up to 2,000 data points per channel (configurable). Older data is discarded to keep memory usage constant.

Chapter 9

Works Offline Too

Once loaded, the app works without internet — and you can install it like a native app. That's the power of Progressive Web Apps.

service-worker.js — The Offline Guardian

A service worker is a script that runs in the background, separate from the web page. It intercepts every network request and can serve cached copies when offline.

// Install: cache all static files self.addEventListener("install", (event) => { event.waitUntil( caches.open(CACHE_VERSION).then(cache => cache.addAll([ "./index.html", "./js/app.js", "./js/config.js", "./styles/main.css", /* ... all files ... */ ]) ) ); }); // Fetch: try network first, fall back to cache self.addEventListener("fetch", (event) => { event.respondWith( fetch(request) .then(response => { // Got it! Cache for next time cache.put(request, response.clone()); return response; }) .catch(() => { // Offline! Serve from cache return caches.match(request); }) ); });

Two-phase offline strategy:

Install When you first visit, the service worker downloads and caches every file the app needs.

Fetch On subsequent visits, it uses a network-first strategy:

  1. Try to fetch from the network (get latest version)
  2. If successful, update the cache
  3. If offline, serve the cached copy

The cache version includes today's date, so it auto-invalidates daily — ensuring you always get updates when online.

📲 Install as an App

The manifest.json file tells the browser "this website can be installed." It defines the app name, icons, theme color, and display mode. When Chrome detects a manifest + service worker, it offers an "Install" prompt.

Once installed, the app gets its own window (no browser toolbar), its own icon in the taskbar, and launches instantly — just like a native desktop app.
